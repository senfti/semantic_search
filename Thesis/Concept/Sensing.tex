\section{Image Processing}

This section describes the extraction of the information needed in later stages of the system from the input images. It covers the object detection, the room type classification and the doorway detection.

\subsection{Object detection}

The object detection takes the images from the main RGBD-camera and its task is to find objects visible in the image and also to estimate their positions in space. In this work the goal is to find objects of a given type, so a general object detection system is needed. Visual object detection is the way to go at the moment. On the one hand visual features are probably the most descriptive features for object detection. On the other hand a lot of research was done on visual object detection and huge datasets containing labeled images are available to train the proposed object detectors. This resulted in the idea to use an object detector working on 2D images and then projecting the object detections into 3D space using the depth image. 

Convolutional Neural Networks(CNNs) are the best performing systems for visual object detection at the moment. The data used to train those CNNs is very important for the accuracy of the trained CNN and also defines the object types the CNN can detect. The COCO-dataset is the best choice at the moment as the number of object types is sufficiently high for this work, but still low enough to not overwhelm an object detector. Table \ref{detectable objects} shows a list of the object types in the COCO-dataset. It also contains enough images, so there is enough variation in the objects to achieve a good generality. The drawback is that objects are not labeled per pixel, but only bounding boxes are given. Therefore the detected objects are also only specified by bounding boxes. As no suitable dataset with objects labeled per pixel is available, the object detection and the later stages of the system have to cope with this problem.

An objectness value, specifying the probability of an detectable object being represented by the bounding box, is attached to every bounding box. Further the probabilities for all object classes are also attached to the bounding boxes. The sum of those probabilities equals the objectness value. To improve the result non-maximum-suppression is applied, to remove duplicated bounding boxes, and also thresholding is done to discard very unlikely bounding boxes.

The result of such a object detection CNN is a set of bounding boxes, where each bounding box has also probability values for each detectable object class. Figure \ref{obj detection} shows a typical result of the object detection, only showing the most likely detections. As one can see the result is far from perfect, even with state-of-the-art object detection CNNs. Many objects are not detected or have a low confidence. Another problem visible in the image is the possibility of overlapping bounding boxes. These problems have to be taken into account in the design of the object mapping system, as accumulation of information from multiple images is needed for more reliable results.

Together with the RGB-image the depth values for every pixel are measured. Due to technical limitations these need not be valid for all pixels. Every pixel with a valid depth value can be projected into 3D space using the intrinsic parameters of the camera. For further processing a sufficient number of pixels with valid depth values are drawn randomly from the image and object probabilities for every object class are assigned to the drawn pixels. Assigning probabilities to pixels based on the bounding boxes of the detections is not possible without further information or assumptions, as the bounding boxes contain pixels not belonging to the object and also overlapping bounding boxes are possible. Segmentation of the pixels in a bounding box into object and background would be possible. However this was not done as this would be quiet time consuming for large numbers of bounding boxes and wrong segmentations are possible, especially with overlapping bounding boxes, occlusion and complex objects. Therefore simply the maximum probability of all bounding boxes overlapping a pixel for an object type is used. The idea behind this is the assumption that all bounding boxes with lower probabilities are just accidentally overlapping. The unreliability of this approach is then taken into account in the inverse sensor model of the object mapping. The final result of the object detection is a set of 3D points, where every point also hold a probability value for all object types.

\subsection{Room type classification}

There are various features which can be used to classify rooms into categories used by humans. In this work this is done solely using visual input. Visual features are very expressive for room classification and also image datasets containing lots of labeled images exist. Other features like shape or size are not used because of their limited expressiveness and the limited data available. The MIT Places-205 Dataset contains about 2.5 million images assigned to 205 place categories and CNNs trained on this dataset are publicly available. Ignoring outdoor place categories this leaves 60 indoor room types, which are listed in table \ref{room types}. The result of the CNN is a probability distribution over the 205 place categories. The outdoor categories are ignored by setting their probabilities to zero. The remaining probabilities are then normalized to one, resulting in a probability distribution over the 60 indoor categories for the room type seen in the input image.

\subsection{Doorway detection}

The detection of doorways is important for the segmentation of the environment into rooms. As there is usually few stuff at the ceiling and the standard height of a doorway is 2 meters, doorways are easily and reliably detected in a depth image of an upward looking RGBD-camera. A passable doorway is detected if 
\begin{itemize}
 \item there is a rectangular surface at about 2 meters height 
 \item this surface has a typical width and depth for a door frame
 \item the area beside the door is occupied, as there has to be a wall
 \item the area in front of and behind the door is free of obstacles, so the robot can drive through
\end{itemize}
These requirements are visualized in Figure \ref{door detection}.

\begin{figure}
  \includegraphics[width=0.9\textwidth,trim=20 180 50 125,clip]{figures/door_detection.pdf}
  \caption{A successfully detected door marked by the purple arrow in the coordinate frame of the robot; black dots are points at about 2 meters height detected by the RGBD-camera, the yellow points are range measurements from the LIDAR, the detected door frame is marked with the blue rectangle, the red rectangles have to be free of LIDAR range measurements and the green rectangle has to contain LIDAR range measurements }
  \label{door detection}
\end{figure}

The position of the detected doorway is set to the center of the rectangle. As can be seen in the hardware section, the camera can only detect doorways in front of the robot. Therefore the orientation of detected doorways is along the smaller axis of the rectangle and pointing away from the robot, so that doorways are always oriented facing outwards.
